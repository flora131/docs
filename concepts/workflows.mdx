A workflow is **one TypeScript file**, versioned with your repo. It encodes a process ‚Äî review, CI, PR, approval, merge; or research, plan, implement, debug ‚Äî as code your team runs identically across machines, agents, and CI.

```ts
import { defineWorkflow } from "@bastani/atomic-sdk/workflows";

export default defineWorkflow({
  name: "review-to-merge",
  description: "Review ‚Üí CI ‚Üí PR ‚Üí Notify ‚Üí Approve ‚Üí Merge",
})
  .for("claude")
  .run(async (ctx) => {
    const review = await ctx.stage({ name: "review" }, {}, {}, async (s) => {
      await s.session.query("Review uncommitted changes for correctness, security, style.");
      s.save(s.sessionId);
    });

    await Promise.all([
      ctx.stage({ name: "security-scan" }, {}, {}, async (s) => {
        await s.session.query("Run `bun audit` and scan for leaked secrets.");
        s.save(s.sessionId);
      }),
      ctx.stage({ name: "ci-checks" }, {}, {}, async (s) => {
        await s.session.query("Run `bun lint` and `bun test`. Report failures.");
        s.save(s.sessionId);
      }),
    ]);

    await ctx.stage({ name: "notify-and-merge" }, {}, {}, async (s) => {
      const t = await s.transcript(review);
      await s.session.query(`Read ${t.path}. Open a PR summarizing the changes.`);
      await s.session.query(
        "Ask the user to confirm approval, then merge with `gh pr merge --squash`.",
        { allowedTools: ["Bash", "Read", "AskUserQuestion"] },
      );
      s.save(s.sessionId);
    });
  })
  .compile();
```

## Stages

Each `ctx.stage(opts, clientOpts, sessionOpts, fn)` call spawns a fresh agent session at runtime ‚Äî its own tmux window, its own context window, and its own node in the execution graph. Inside the callback you write raw provider SDK code against `s.session` / `s.client`. A stage callback can return a value, which becomes `handle.result` on the returned `SessionHandle` ‚Äî that's how an LLM's verdict drives TypeScript control flow.

See [stages](/sdk/stages) for the full callback API.

## The execution graph

The topology is **auto-inferred** from your `await` / `Promise.all` patterns ‚Äî no annotations. Sequential `await` creates a chain; `Promise.all` creates parallel fan-out and fan-in. Use ordinary TypeScript control flow ‚Äî `for`, `if/else`, `try/catch` ‚Äî and the graph follows.

- **Branch** ‚Äî `if`/`else` on a prior stage's `result`.
- **Retry / loop** ‚Äî a bounded `for` loop with early exit when a review comes back clean (see the [`review-fix-loop`](/workflows/built-in) pattern).
- **Parallel** ‚Äî `Promise.all([ctx.stage(...), ctx.stage(...)])`; parallel siblings finish (or fail fast) before the next stage starts.
- **Nested** ‚Äî call `s.stage(...)` inside a callback to spawn child sessions, shown as nested graph nodes.

Atomic renders a live workflow panel over the session graph as the run executes ‚Äî nodes per stage with status, edges for dependencies, pane previews, and highlighted transcript hand-offs.

## Session isolation

Every stage runs in its own session with a fresh context. There's no shared mutable state between stages ‚Äî data flows only through explicit transcript reads.

## Controlled transcript hand-offs

A stage calls `s.save(...)` to persist its output. A later stage reads it with `s.transcript(handle)` (by handle) or `s.transcript("stage-name")` (by name), getting `{ path, content }`. Read transcripts only come from **completed** upstream stages ‚Äî parallel siblings can't read each other.

This is token-aware by design: save full transcripts to disk and pass paths or distilled summaries forward instead of stuffing every stage with the entire history.

```ts
const describe = await ctx.stage({ name: "describe" }, {}, {}, async (s) => {
  await s.session.query("Describe this project.");
  s.save(s.sessionId);
});

await ctx.stage({ name: "summarize" }, {}, {}, async (s) => {
  const research = await s.transcript(describe);
  await s.session.query(`Read ${research.path} and summarize in 2-3 bullets.`);
  s.save(s.sessionId);
});
```

## Headless stages

Pass `headless: true` and the stage runs the provider SDK in-process instead of spawning a tmux window ‚Äî invisible in the graph, tracked by a background counter in the statusline. The callback API is identical. Useful for fan-out data gathering where you don't need a visible pane per worker.

## Human-in-the-loop gates

A stage can pause for a human. Issue a query that asks the user a question (e.g. with `AskUserQuestion` in the allowed tools) and the stage blocks until they respond ‚Äî then the workflow continues or branches on the answer. The same gate works inside a headless stage.

## Determinism

`.compile()` freezes the workflow ‚Äî step order, session names, and the execution graph become immutable. Step 2 never starts until Step 1 finishes. The same definition produces the same execution order with the same data flow, anywhere. Variance comes from the LLM's responses, not from a changing workflow.

Every stage writes its messages, transcript, and metadata to disk under `~/.atomic/sessions/<runId>/` ‚Äî a complete, inspectable execution record. See [reading run artifacts](/workflows/running#reading-run-artifacts).
