# Authoring workflows

Custom workflows are TypeScript files that export a workflow definition. They are the right abstraction when your team has a repeated process: research before implementation, review loops, migration playbooks, release checks, or incident-response automation.

## Where workflows live

Project workflows should live in a checked-in workflow directory, commonly:

```text
.atomic/workflows/
```

User-level workflows can live in the user configuration directory. Packages can also contribute workflows.

## Start with the workflow skill

The fastest way to create a correct workflow for your installed Atomic version is to ask Atomic itself:

```text
Use the workflow skill to scaffold a workflow named review-to-merge. It should take --issue and --max-rounds inputs, research the issue, implement the change, run review agents, and pause before final merge instructions.
```

The skill knows the current SDK conventions and can write the file into your project.

## Workflow shape

A workflow usually has:

1. **Metadata** — name, description, and optional grouping.
2. **Inputs** — typed CLI/TUI fields such as `prompt`, `issue`, `maxLoops`, or `dryRun`.
3. **Stages** — deterministic orchestration steps.
4. **Handoffs** — summaries or artifacts passed between stages.
5. **Validation and gates** — commands, checks, approvals, and stop rules.

```ts
import { defineWorkflow } from "@bastani/atomic/sdk";

export default defineWorkflow({
  name: "research-then-implement",
  description: "Research a task, implement it, then run a focused review.",
  inputs: {
    prompt: {
      type: "string",
      required: true,
      description: "The task to research and implement.",
    },
    maxReviewRounds: {
      type: "number",
      default: 1,
      description: "Maximum review/fix cycles.",
    },
  },
  async run(ctx) {
    const research = await ctx.task("research", {
      prompt: `Map the codebase for this task. Return files, constraints, and risks.\n\n${ctx.input.prompt}`,
    });

    const implementation = await ctx.task("implement", {
      prompt: `Implement the task using this research. Return changed files and validation evidence.\n\n${research.text}`,
    });

    await ctx.task("review", {
      prompt: `Review the implementation for correctness and regressions.\n\n${implementation.text}`,
    });
  },
});
```

## Parallel fan-out

Use parallel stages when work is independent:

```ts
const [fileMap, patterns, priorResearch] = await ctx.parallel([
  ctx.task("locate", { prompt: "Find files related to auth middleware." }),
  ctx.task("patterns", { prompt: "Find similar middleware implementations." }),
  ctx.task("history", { prompt: "Find prior research docs about auth." }),
]);
```

Good parallel work:

- codebase discovery
- pattern search
- independent reviews
- external documentation research
- validation checks that do not mutate the same files

Avoid parallel writers in the same worktree unless each writer is isolated.

## Human-in-the-loop gates

Use UI gates for decisions the model should not make alone:

```ts
const decision = await ctx.ui.ask({
  title: "Proceed with migration?",
  message: "Research found a schema change. Continue to implementation?",
  options: ["Continue", "Stop and summarize"],
});

if (decision !== "Continue") return;
```

Use gates for:

- scope expansion
- destructive commands
- unclear product decisions
- risky migrations
- final approval before publishing or merging

## Model fallbacks

Workflows can define preferred models and fallback chains per stage. Use this for reliability: pick the strongest model for planning/review and a faster/cheaper model for search or formatting. See [Workflow SDK and runtime reference](/workflows/workflow-reference) for the current option names.

## Output discipline

Ask every writer stage to return:

- changed files
- what was implemented
- what was intentionally left undone
- commands run and exit codes
- validation evidence
- risks or surprises
- decisions that need human approval

This makes downstream review stages much more reliable.

## Testing a workflow

1. Run it on a tiny prompt first.
2. Inspect artifacts and session outputs.
3. Validate pause/resume paths.
4. Confirm failures are visible and actionable.
5. Commit the workflow with a README or usage comment.

## Common workflow patterns

| Pattern | Shape |
| --- | --- |
| Research handoff | locate/analyze/patterns in parallel → write research doc |
| Implementation loop | research → implement → review → fix → validate |
| Migration | inventory → shard plan → per-shard implementation → global validation |
| Release prep | changelog → tests → docs → approval gate |
| Design handoff | discover design system → generate → refine → export artifacts |
