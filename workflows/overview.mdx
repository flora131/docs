# Workflows

Workflows are Atomic's answer to “how do we make this agent process repeatable?” A workflow is a TypeScript definition that launches one or more agent stages, passes structured context between them, gates progress on human input when needed, and records execution state so long-running work can be inspected or resumed.

## Why workflows exist

A single coding-agent session is flexible, but process lives in the prompt. Workflows move the process into code:

- the same stages run every time
- parallel work is explicit
- context handoffs are deliberate
- human approvals are part of the graph
- model choices and fallbacks are configured, not improvised
- long-running jobs have status, artifacts, and resumability

## Built-in workflows

| Workflow | Use it for | Typical command |
| --- | --- | --- |
| `deep-research-codebase` | Broad repository research written to `research/docs/` for future specs and workflows. | `/workflow deep-research-codebase prompt="Map the billing system"` |
| `ralph` | Autonomous plan → implement → simplify → review loops for bounded coding tasks. | `/workflow ralph prompt="Implement the cache invalidation spec"` |
| `open-claude-design` | Design generation and handoff from a prompt/reference asset. | `/workflow open-claude-design prompt="Team activity feed" output_type=prototype` |

Run:

```text
/workflow list
/workflow inputs <name>
```

## The authoring model

A workflow is authored with `defineWorkflow()`:

```ts
import { defineWorkflow } from "@bastani/atomic/sdk";

export default defineWorkflow({
  name: "review-change",
  description: "Research, implement, and review a focused change.",
  inputs: {
    prompt: { type: "string", required: true },
  },
  async run(ctx) {
    const research = await ctx.task("research", {
      prompt: `Research the codebase for: ${ctx.input.prompt}`,
    });

    const implementation = await ctx.task("implement", {
      prompt: `Implement the requested change using this research:\n${research.text}`,
    });

    await ctx.task("review", {
      prompt: `Review the implementation for regressions:\n${implementation.text}`,
    });
  },
});
```

  The exact helper names available in your installed version are documented in [Workflow SDK and runtime reference](/workflows/workflow-reference). Use the `workflow` skill in Atomic to scaffold version-correct workflow files.

## Core primitives

| Primitive | What it does |
| --- | --- |
| `ctx.task()` / `ctx.stage()` | Run one agent-backed stage with a prompt and options. |
| `ctx.parallel()` | Fan out independent stages and wait for all results. |
| `ctx.chain()` | Run sequential stages where each output feeds the next. |
| `ctx.ui` | Ask the human questions, present approvals, and collect decisions. |
| Model fallbacks | Try a preferred model, then fall back when unavailable. |
| Artifacts | Persist files, outputs, and run metadata for inspection. |

## Execution surfaces

You can run workflows from:

- the TUI slash command: `/workflow ...`
- the workflow tool exposed to agents
- keyboard/UI shortcuts such as the workflow panel
- TypeScript/SDK entry points in automation

Common lifecycle commands:

```text
/workflow list
/workflow inputs <name>
/workflow <name> key=value
/workflow status <run-id>
/workflow resume <run-id>
```

## Best practices

- Keep each stage narrow and outcome-oriented.
- Pass summarized context forward instead of raw transcripts when possible.
- Use parallel stages for independent research/review, not conflicting writes.
- Put approval gates before irreversible or high-scope actions.
- Return validation evidence from implementation stages.
- Store custom workflows in the repo so the team can review and version them.
